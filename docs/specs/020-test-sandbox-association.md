# Spec 020: Test Sandbox Association

**PRD Reference:** N/A (developer tooling)
**Priority:** Must Have

## What This Feature Does

Creates a "Test Sandbox" association in the production database that clones all NGHA U15 data (players, teams, continuation rounds, scraper configs). The sandbox is hidden from regular users (`join_enabled = false`) and only accessible to admin accounts. This gives developers a safe copy of real data to test features against without risking the live Nepean Wildcats data.

## Changes Required

### Database

One new migration file: `backend/supabase/migrations/20260423000001_create_test_sandbox.sql`

This migration:

1. **Creates the sandbox association** with a fixed UUID and `join_enabled = false`
2. **Clones U15 teams** from NGHA into the sandbox with new UUIDs, preserving name, display_order, and max_roster_size
3. **Clones U15 players** from NGHA (where `deleted_at IS NULL` and `suggested_by IS NULL`) with new UUIDs, remapping `team_id` to the cloned team UUIDs
4. **Clones continuation rounds** for U15 from NGHA with new UUIDs
5. **Clones continuations URLs** for U15 from NGHA with new UUIDs
6. **Clones scraper configs** from NGHA with new UUIDs (scraper configs are association-level, not division-scoped)
7. **Grants access** by inserting `user_associations` rows for all existing NGHA admin/group_admin users with `group_admin` role on the sandbox

#### UUID Remapping Strategy

Use PostgreSQL temp tables to map old IDs to new IDs:

1. **Teams:** Clone with `gen_random_uuid()`. Join on `(division, name)` to build old→new team ID mapping.
2. **Players:** Clone with `gen_random_uuid()`. Use team mapping to remap `team_id` FK. No further downstream remapping needed since user-owned data is not cloned.
3. **Continuation rounds:** Clone with `gen_random_uuid()`. No downstream FKs to remap (continuation_orders are user-owned and not cloned).
4. **Continuations URLs, scraper configs:** Clone with `gen_random_uuid()`. No FKs to remap.

#### Trigger Handling

Disable audit triggers before player inserts, re-enable after (same pattern as `20260421000001`):

```
ALTER TABLE tryout_players DISABLE TRIGGER trg_audit_players_insert;
-- ... INSERT statements ...
ALTER TABLE tryout_players ENABLE TRIGGER trg_audit_players_insert;
```

#### Data NOT Cloned (Fresh Start in Sandbox)

These tables are user-owned or generated during use. Admins build them up naturally while testing:

- `player_predictions` — per-user draft rankings
- `previous_team_orders` — per-user sort orders
- `continuation_orders` — per-user continuation sort orders
- `player_annotations` — per-user notes, custom names, favourites
- `player_hearts` — per-user hearts
- `corrections` — parent-submitted change requests
- `audit_log` — auto-generated by triggers during use
- `user_tracked_groups` — auto-created when user selects a division

## Key Implementation Details

### Sandbox Association Values

| Field | Value |
|-------|-------|
| `id` | `a2000000-0000-0000-0000-000000000002` (deterministic, easy to reference) |
| `name` | `Test Sandbox (U15)` |
| `abbreviation` | `TEST` |
| `join_code` | `TEST2026` |
| `join_enabled` | `false` |

### Conflict Analysis (All Clear)

These items were audited and confirmed safe:

- **Join code:** `TEST2026` does not conflict with existing `NGHA2026`. The `join_code` column has a database-wide UNIQUE constraint — verified no collision.
- **Team names:** Identical names (AA, A) in different associations are fine. Team uniqueness is by UUID, not name. No unique constraint on `(association_id, name)`.
- **Jersey numbers:** Not unique in the database. Same jersey numbers across associations cause no issues.
- **Player names:** Not unique. Same names across associations cause no issues.
- **Division name "U15":** Just a text value, scoped by `association_id` in all queries and constraints.
- **Continuation rounds:** Unique constraint is `(association_id, division, team_level, round_number)` — includes `association_id`, so safe.
- **Continuations URLs:** Unique constraint is `(association_id, division)` — includes `association_id`, so safe.
- **RLS policies:** Migration runs as postgres superuser, bypasses all RLS. Post-migration, RLS will isolate sandbox data to sandbox members only.
- **Enum values:** All cloned statuses use existing valid enum values (copying directly from source).

### Scope Guard: U15 Only

Every SELECT in the migration must include `AND division = 'U15'` (for division-scoped tables) or filter by the NGHA association ID. The only exception is `scraper_configs`, which is association-level (not division-scoped) — clone all configs so the scraper works in the sandbox.

### Admin Access

The migration auto-grants access by querying existing NGHA admins:

```
INSERT INTO user_associations (user_id, association_id, role)
SELECT user_id, 'sandbox-uuid', 'group_admin'
FROM user_associations
WHERE association_id = 'a1000000-...'
  AND role IN ('admin', 'group_admin');
```

This ensures every current NGHA admin automatically gets sandbox access. No hardcoded user IDs needed.

### How to Remove the Sandbox Later

If you ever want to remove the sandbox, a single DELETE cascades everything:

```sql
DELETE FROM associations WHERE id = 'a2000000-0000-0000-0000-000000000002';
```

All teams, players, continuation rounds, URLs, configs, user_associations, and any user-generated data (predictions, annotations, hearts, orders) cascade-delete automatically via `ON DELETE CASCADE` foreign keys.
